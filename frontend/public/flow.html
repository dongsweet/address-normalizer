一次地址处理会经过哪些路径

1

清洗输入

去掉配送备注、电话、口语化噪声；保留真正可能影响地址定位的信息。

2

拆锚点和明细

把“小区/门店/园区”与“18栋1单元2层8号”拆开，避免用房号污染召回。

3

本地候选召回

优先查授权 POI、记忆库、标准库，命中高置信候选时直接跳过慢模型。

4

地图API补召回

本地找不到时再调地图 API，限制城市为乌鲁木齐，减少跨城市误召回。

5

Qwen判别

Qwen 不负责发明地址，只在候选里选最可信的一条，或者拒识。

6

输出和沉淀

输出 JSON/单行地址；主记忆只存小区/POI锚点，房号明细单独沉淀。

为什么 POI 数据是核心

POI 是“现实世界锚点”

外卖地址经常写“美美一楼”“云计算中心”“某小区东门”，这些不是门牌号，而是 POI 或商圈锚点。

地图 API 适合补漏

API 能补召回最新点位，但不适合把每个结果都当成本地数据库长期保存，授权边界要单独确认。

授权 POI 适合做底座

可本地索引、可批量召回、可审计版本；需要确认是否允许存储、检索、派生字段和内部服务使用。

记忆库提升复用

客户业务里反复出现的地址写法会沉淀下来，下一次直接命中，减少地图 API 和大模型调用。

系统里的几类数据

数据层 现在怎么用 对客户的价值 授权/风险点
公开POI快照 Demo 里用于乌鲁木齐 POI 抽样召回。 低成本证明流程，但覆盖率、鲜度、中文地址质量有限。 适合演示和技术验证，不建议直接承诺生产覆盖率。
授权POI库 未来作为主召回底座，存 POI 名称、地址、分类、经纬度、行政区、版本号。 提升命中率，减少地图 API 调用，支撑离线批处理和审计。 采购合同必须写清：本地存储、内部检索、更新频率、派生数据和脱敏沉淀权限。
地图API 本地召回不足时补召回；Demo 可配置可关闭。 处理新开业、新改名、新园区等动态变化。 注意调用量、缓存期限、结果落库范围和服务条款。
业务记忆库 只存小区/POI锚点，不按房号膨胀。 让客户自己的地址写法越用越准，且成本可控。 应保留来源、命中次数、确认人和时间，便于回滚。
房号明细库 单独记录楼栋、单元、楼层、房号的原始写法和规范写法。 后续可做“小区级房号写法规范化”，例如把“1-202”“1单202”“1单元202”统一。 避免把明细当锚点召回，防止旧房号和新房号串联。
标准地址库 当前可选；拿到后作为最终标准 ID 对齐层。 能输出标准地址 ID、行政区划、门楼牌，适合和公安/政务库做强匹配。 通常获取难、授权严，但一旦有，置信度和合规价值最高。

POI 库该怎么买，预算怎么说

建议采购形态

先按“乌鲁木齐城市级 POI 快照 + 定期更新 + 少量地图 API 校验包”采购，不建议一开始就买全国全量。 数据字段至少包括：POI ID、名称、别名、地址、经纬度、分类、区县/街道、更新时间、上下线状态、供应商版本号。

合同里重点看三句话：是否允许本地存储，是否允许内部检索服务使用，是否允许把客户确认后的锚点和脱敏明细沉淀为自有记忆。

预算口径

公开页面更多展示 API 计费、配额或企业服务入口，完整 POI 数据库通常需要商务询价。 给内部预算时，可以先按“单城市试点、年度授权、定期更新、API 调用包”询价；全国范围、实时更新、高并发 SLA 会明显增加成本。

对外表述建议：Demo 使用公开 POI 快照和可选地图 API；生产建议采购授权 POI 库。具体金额以供应商正式报价为准，我们先按城市级试点做预算和合同条款评估。

标准库能带来什么

从“像这个地址”变成“就是这个地址”

POI 能帮我们找到锚点，标准地址库能给出最终标准 ID，这是匹配公安/政务数据库的关键。

置信度更可解释

有标准库时，结果可以说明命中了哪条标准记录、哪个行政区、哪个门楼牌。

减少地图依赖

高频地址进入标准库或记忆库后，地图 API 只处理新增和疑难样本。

支持后续治理

可统计小区常见房号写法、缺失街道、异常坐标、拒识地址，形成数据治理闭环。
